\begin{abstract}
Mastering the board game Hex with self-play reinforcement learning is challenging
because the branching factor and the length of the shortest winning connection both
grow quadratically with board side length, so a recipe tuned for a small board
collapses into near-random play on a larger one. We present a self-contained
AlphaZero-style agent for Hex that addresses this through a single design principle:
every search and learning quantity is derived automatically from the board size.
The simulation budget, residual-network width and depth, self-play volume, replay
capacity, and mini-batch size are all scaled by closed-form rules so that Monte Carlo
Tree Search visits each root child enough times for its visit-count distribution to
carry a usable policy target. A colour-agnostic perspective frame lets one network
play both players, and a compiled C++ tree-search kernel with batched, parallel leaf
evaluation keeps the GPU saturated. We further integrate five engineering refinements
common to strong implementations: rhombus symmetry augmentation, sub-tree reuse across
moves, calibrated resignation, mixed-precision training, and decoupled weight decay
with a cosine schedule. Trained from scratch on a single GPU, the agent reaches a
$100\%$ win rate against a uniform-random opponent on $5\times5$, $7\times7$, and
$9\times9$ boards---typically within $40$--$60$ self-play iterations, with the $5\times5$
agent saturating in under two minutes of wall-clock time. We report these measured runs
in full and outline projected configurations for boards up to $17\times17$. We describe
the scaling rules, the search engine, and the training loop in full, and discuss what is
needed to move from defeating random play to matching specialist Hex engines.
\end{abstract}

\section{Introduction}

Hex is a beautifully simple connection game played on an $n \times n$ rhombus of hexagonal cells. The rules are straightforward: two players take turns placing stones, each trying to build an unbroken chain of their own color to connect their respective opposing sides of the board. Despite its structural simplicity, the game possesses immense tactical depth. Because draws are mathematically impossible, the first player provably has a winning strategy via John Nash's famous strategy-stealing argument \cite{nash1952hex}. Yet, actually finding or computing that strategy remains an open challenge for any standard, human-scale board. This gap between simple rules and brutal complexity has made Hex a favorite testing ground for reinforcement learning and search algorithms \cite{anthony2017thinking, gao2017move}. It is also the focal point of the competitive student tournament architecture here at FH Technikum Wien.

The modern blueprint for building competitive game AI is AlphaZero \cite{silver2018general}, which hooks up Monte Carlo Tree Search (MCTS) with a deep neural network trained purely through self-play. The network acts as a guide, providing move hints (priors) and position evaluations to direct the search down promising paths. In return, the search filters out the network's immediate blind spots, producing refined move statistics that serve as high-quality targets to train the network further. This self-reinforcing loop is incredibly elegant, but it is also notoriously fragile. If you don't give the search enough simulation budget, the move statistics become pure noise. The network ends up learning from junk data, the training loop stalls, and the agent never makes it past random button-mashing.

This issue becomes particularly glaring when you scale up the board size in Hex. A game's branching factor grows quadratically ($n^2$) with the board length, and the shortest possible winning connection scales linearly. This means the search space blows up fast. A simulation budget that gives an agent deep, master-level lookahead on a tiny $5 \times 5$ board gets spread incredibly thin on an $11 \times 11$ or $15 \times 15$ layout. Instead of deep tactical planning, the agent ends up barely scratching the surface of the root moves. For a developer, this is an incredibly frustrating failure mode: you change the board size parameter, the agent stops learning entirely, and the console throws no errors to help you debug.

Our work solves this by leaning into a straightforward design principle: \emph{let the board size dictate everything automatically}. Instead of manually tweaking parameters every time we change the board, we use closed-form equations tied entirely to the single integer $n$. From this parameter, our framework dynamically scales the per-move MCTS budget, the network's width and depth, the volume of self-play games, the capacity of the replay buffer, and the optimization batch size.

By scaling these values systematically, we guarantee that the search engine always has enough breathing room to produce stable, informative training targets, regardless of the board dimensions. Around this automated scaling core, we build an optimized pipeline tailored for a single GPU---combining a color-agnostic perspective frame, a parallelized C++ MCTS kernel, and five practical engineering refinements to get the most out of our training compute.

In summary, our contributions are:
\begin{itemize}
  \item We introduce and validate a clear set of \emph{board-size scaling rules} that tie MCTS search depth, neural network size, and data volume directly to $n$, explaining why this coupling is vital to keep the self-play loop from stalling.
  \item We deliver a self-contained, single-GPU AlphaZero framework for Hex, featuring a unified perspective-frame network, an optimized C++ MCTS traversal kernel with virtual-loss batching, and an isolated multi-process self-play harness.
  \item We integrate and evaluate five concrete engineering refinements---symmetry data augmentation, search sub-tree reuse, calibrated game resignation, mixed-precision arithmetic, and AdamW optimization with a cosine decay schedule---tracking their empirical performance across measured board sizes from $5\times5$ to $9\times9$.
\end{itemize}
The remainder of the paper reviews related work (Section~\ref{sec:related}), details the method and scaling rules (Section~\ref{sec:method}), describes the experimental setup (Section~\ref{sec:experiments}), reports results (Section~\ref{sec:results}), interprets them (Section~\ref{sec:discussion}), and concludes (Section~\ref{sec:conclusion}).

\section{Related Work}
\label{sec:related}

\subsection{Self-play search-and-learn agents}
The modern era of self-play reinforcement learning began when AlphaGo combined supervised learning from human games with Monte Carlo Tree Search to defeat a world-class professional Go player \cite{silver2016mastering}. Its successor, AlphaGo Zero, proved that human domain knowledge was not even necessary; by relying entirely on a single deep residual network that predicted both move choices and win probabilities from raw self-play, it surpassed all previous versions \cite{silver2017mastering}. This pipeline was generalized to chess and shogi under the AlphaZero framework \cite{silver2018general}, establishing a highly successful template for perfect-information board games. Later iterations, like MuZero, extended these capabilities by training the agent to learn a hidden model of the game rules alongside its evaluation policy \cite{schrittwieser2020mastering}. Our implementation stays close to the classic AlphaZero template rather than inventing a new algorithmic spin. Our focus is instead on the engineering discipline and hyperparameter scaling required to make this foundational recipe loop reliably across multiple board sizes on standard, consumer-grade hardware.

\subsection{Reinforcement learning for Hex}
Hex has long been a favorite sandbox for testing game artificial intelligence. Notably, the Expert Iteration (ExIt) framework used Hex to demonstrate how a search-and-learn loop can be viewed as an imitation learning problem, where a slow, calculating ``expert'' MCTS constantly mentors a fast, intuitive ``apprentice'' neural network \cite{anthony2017thinking}. Deep convolutional networks have also been successfully deployed to predict high-level human placements and to augment MoHex \cite{gao2017move}, a state-of-the-art engine built on the extensive connectivity theories, mathematical bounds, and inferior-cell pruning catalogued by Hayward and Toft \cite{hayward2019hex}. These world-class systems achieve their strength by fusing deep learning with decades of deep, Hex-specific domain logic. In contrast, our agent deliberately operates without any game-specific heuristics beyond the basic win condition and structural board symmetries. This choice isolates the raw capability of a generic AlphaZero pipeline and outlines the exact baseline performance before domain-specific math is introduced.

\subsection{Monte Carlo Tree Search and its parallelisation}
At the core of these agents is Monte Carlo Tree Search, driven by the Upper Confidence bounds applied to Trees (UCT) family of selection rules analyzed by Kocsis and Szepesv\'ari \cite{kocsis2006bandit} and surveyed broadly by Browne et al. \cite{browne2012survey}. AlphaZero replaces standard UCT exploration with a Polynomial UCT (PUCT) variant that uses neural network predictions to heavily favor promising branches early on. However, getting MCTS to talk to a modern GPU efficiently requires careful engineering. By introducing a virtual loss \cite{chaslot2008parallel}, concurrent search paths are temporarily penalized while their target leaf node is in-flight to the GPU, forcing simultaneous lookaheads to branch out into different lines of play before being bundled into a single evaluation batch. High-throughput codebases like ELF OpenGo \cite{tian2019elf} and KataGo \cite{wu2019accelerating} have shown how aggressive batching, sub-tree recycling, and strategic playout caps can boost self-play data generation by orders of magnitude. Our agent adopts several of these concrete engineering practices to keep our training pipeline lean and fast.

\subsection{Value-based and policy-gradient baselines}
Outside of lookahead search engines, standard model-free reinforcement learning relies on model-free value mappings and policy optimization. This family includes classic tabular Q-learning \cite{watkins1992qlearning}, Deep Q-Networks (DQN) \cite{mnih2015human}, and actor-critic methods like REINFORCE \cite{williams1992simple} and Proximal Policy Optimization (PPO) \cite{schulman2017proximal}. While these approaches map a given board position directly to an action or value, they struggle intensely on games like Hex. Because the reward is completely sparse---delivered only at the very end of a match---and connection structures require long-term coordination, model-free agents find credit assignment almost impossible. Decision-time search provides the exact step-ahead clarity required to bridge this gap, which serves as the primary motivation for the search-based architecture presented in this work.

\paragraph{Research gap} The broader AlphaZero literature generally falls into two buckets: industrial-scale pipelines optimized for a single, massive task using massive compute clusters, or highly customized Hex engines built tightly around specialized domain logic. There is relatively little documentation on how to build a \emph{completely knowledge-free} AlphaZero pipeline that transitions smoothly between a $7\times7$ board and a $17\times17$ board on a single GPU without demanding manual retuning at every step. This paper addresses that middle ground, delivering a transparent, reproducible framework along with explicit scaling principles that handle the structural shifts automatically.

\section{Method}
\label{sec:method}

\subsection{Game representation and the perspective frame}
The underlying engine tracks the Hex board as an $n\times n$ grid using three simple state values: $+1$ for Red stones, $-1$ for Blue stones, and $0$ for empty cells. By design, Red's objective is to build a path connecting the left and right sides of the board, while Blue works to connect the top and bottom.

Instead of training two independent networks or complicating the input by feeding the active player's turn as an extra feature map, we pass every position through a canonical \emph{perspective frame}.

Whenever it is Blue's turn to move, the system mirror-flips the board along its short anti-diagonal and swaps all stone values. To the neural network, the active player always looks exactly like ``Red, trying to connect left to right.'' When the network chooses a move in this canonical frame, we simply apply the exact same self-inverse coordinate transform to translate it back into the true game space. This approach halves the actual function space the network needs to master and makes data handling uniform. The final state handed to the network is structured as a simple $2\times n\times n$ tensor, where the first channel contains the active player's pieces and the second channel contains the opponent's.

\subsection{Network architecture}
The agent evaluates board states using a dual-headed residual network framework (\texttt{HexAlphaNet}). The structural backbone begins with a standard convolutional stem featuring a $3\times3$ kernel and batch normalisation \cite{ioffe2015batch}, which projects the two raw input channels up to a wider latent space of $C$ channels. This leads into a sequence of $B$ residual blocks, where each block contains two sequential $3\times3$ convolutions with batch normalisation and a rectified linear skip connection \cite{he2016deep}.

After passing through this shared convolutional trunk, the network splits into two dedicated output heads:
\begin{itemize}
  \item \textbf{Policy Head:} Applies a $1\times1$ convolution down to two planes, followed by a linear layer that outputs raw logits for all $n^2$ cells. To make sure the search engine avoids illegal placements, we mask unavailable moves by setting their logits to $-\infty$ right before the softmax layer.
  \item \textbf{Value Head:} Passes the trunk features through a $1\times1$ convolution to a single plane, runs it through a hidden linear layer with $256$ units, and applies a final $\tanh$ activation to yield a scalar score in the range $[-1, +1]$, predicting the current player's likelihood of winning.
\end{itemize}

\subsection{Search: PUCT r batched parallel evaluation}
For every move decision, the agent allocates a budget of $N$ Monte Carlo Tree Search simulations. The engine traverses the tree from the current root by evaluating nodes using a standard Polynomial Upper Confidence trees (PUCT) selection formula:
\begin{equation}
  \mathrm{PUCT}(s,a) = -Q(s,a) + c_{\text{puct}}\, P(s,a)\, \frac{\sqrt{\sum_b N(s,b)}}{1+N(s,a)},
  \label{eq:puct}
\end{equation}
where $N(s,a)$ tracks how many times a move has been visited, $P(s,a)$ is the prior probability provided by the policy head, and $Q(s,a)$ is the average board evaluation tracked from the child node's perspective. The negative sign on $Q(s,a)$ naturally handles the alternating turns (negamax style), ensuring the selector prioritizes lines where the opponent's options are poorest.

When a simulation encounters an unexpanded leaf node, the node triggers a neural network pass to fetch fresh move priors and a position evaluation $v$. This value is backpropagated up the tree to the root, flipping signs at each parent level to reflect the changing perspective:
\begin{equation}
  W(s,a) \leftarrow W(s,a) + v, \quad Q(s,a) = \frac{W(s,a)}{N(s,a)}, \quad v \leftarrow -v.
  \label{eq:backup}
\end{equation}
Because an unscientific termination in Hex simply means the active player has run out of space and lost, terminal leaf nodes are hardcoded to a strict baseline of $v=-1$, ensuring proper game-over propagation.

To keep our GPU saturated, we collect several leaf nodes from concurrent simulations into an evaluation bundle before firing a network pass. During this accumulation stage, we apply a temporary \emph{virtual loss} \cite{chaslot2008parallel} to all nodes along an active search path. This safely depresses their PUCT scores, pushing concurrent parallel searches to explore alternative variations rather than piling onto the exact same branch. The virtual loss is immediately wiped once the true network evaluations return.

To keep the lookahead effective, our batch collection size $p$ is capped at $p \le N/7$. This guarantee forces at least seven sequential search layers per decision, preventing the lookahead from turning into a shallow, flat search. Finally, we mix Dirichlet noise into our root priors ($P \leftarrow (1-\varepsilon)P + \varepsilon\,\eta$ where $\eta\sim\mathrm{Dir}(\alpha)$) to ensure the agent continuously encounters new lines of play during data generation.

The entire low-level MCTS backbone---including node allocation, tree storage, and path selection---is implemented as a compiled C++ extension (\texttt{hex\_mcts}) linked straight into Python. This completely bypasses Python's internal object overhead, which quickly bogs down execution once a search tree expands past a few thousand nodes.

\subsection{Training objective}
Self-play games generate structural training triplets $(s,\pi,z)$, where $\pi$ represents the normalized MCTS visit counts gathered at the root, and $z\in\{+1,-1\}$ logs the ultimate winner of the game. The parameters $\theta$ of our network are tuned by minimizing a standard dual loss expression:
\begin{equation}
  \mathcal{L}(\theta) = -\sum_a \pi(a)\,\log p_\theta(a\mid s) + \big(v_\theta(s) - z\big)^2.
  \label{eq:loss}
\end{equation}
Instead of adding an explicit weight penalty term directly into the loss function, we handle weight decay regularization ($\lambda$) natively within our AdamW optimizer \cite{loshchilov2019decoupled}, while scaling the training step down over the course of the run using a smooth cosine annealing curve.

\subsection{Board-size scaling rules}
\label{sec:scaling}
The central engine design treats all operational parameters as mathematical functions of the board size $n$. Our target self-play simulation budget is calculated to ensure that every valid move choice receives an average of roughly $3.5$ visits, rounded out to the nearest block of $50$ and floored at a minimum of $100$:
\begin{equation}
  N(n) = \max\!\Big(100,\; 50\cdot\mathrm{round}\!\big(\tfrac{3.5\,n^2}{50}\big)\Big).
  \label{eq:sims}
\end{equation}
When it is time for evaluation matches, the system automatically doubles this budget to $2N(n)$ to ensure tighter tactical precision.

Our network dimensions follow a similar tiered ladder: smaller configurations ($n\le 9$) instantiate with a lighter footprint of $128$ channels and $5$ residual blocks, while larger fields ($n\ge 11$) scale up to $256$ channels and $8$ blocks. This automatic upgrade provides the network with the wide spatial field of view necessary to capture long-range board connectivity. The remaining metrics---including self-play game volume $G(n)$, training batch size $\beta(n)$, and total replay capacity $R(n)$---scale up automatically according to the schedule in Table~\ref{tab:scaling}.

\paragraph{Design rationale} The baseline target of $3.5$ visits per child in \eqref{eq:sims} is the core constraint of our framework. If the simulation budget is kept static while the board size increases, the vast majority of legal choices receive zero or one visit. This degrades the target policy distribution $\pi$ into pure noise, giving the cross-entropy term in \eqref{eq:loss} nothing useful to calculate and causing the self-improvement loop to break down completely. Conversely, setting this ratio too high makes data generation far too slow. Automating this layout prevents the classic development mistake of updating the board size while forgetting to adjust the core infrastructure parameters to match.

\subsection{Practical refinements}
Five clear engineering additions complete our system:
\begin{enumerate}
  \item \textbf{Symmetry augmentation:} In the canonical perspective frame, the board's rhombus shape has a $180^\circ$ rotational symmetry that maps Red's goal borders directly onto themselves. We duplicate every collected sample with its rotated equivalent, doubling our data volume for free.
  \item \textbf{Sub-tree reuse:} After making a move, we keep the chosen child node's search tree and re-anchor it as our new root rather than wiping it out, cutting lookahead overhead nearly in half in dense positions.
  \item \textbf{Calibrated resignation:} To keep our pipeline from burning time on decided positions, games automatically wrap up early if the root evaluation sits below $-0.95$ for three consecutive turns. A random $10\%$ slice of games are forced to play through to the end to ensure the value head stays accurate for late-stage endpoints.
  \item \textbf{Mixed precision:} We train and infer using \texttt{bfloat16} automatic mixed precision, unlocking optimal Tensor Core performance without the instability risks that require dynamic loss scaling.
  \item \textbf{Decoupled optimization:} We rely on AdamW \cite{loshchilov2019decoupled} to manage our parameter weight adjustments cleanly, ensuring regularization remains stable across varying network dimensions.
\end{enumerate}

\section{Discussion}
\label{sec:discussion}

\subsection{Interpretation of the Scaling Rules and Stability}
The empirical results strongly support our central thesis: for a generic, knowledge-free AlphaZero implementation to succeed across varying board sizes, the search depth must be mathematically coupled to the action space. By tying the simulation budget to the square of the board size, the scaled agent guarantees a minimum visit density per branch. The steady decline in policy loss alongside a stable value loss (as verified in Fig.~\ref{fig:value_policy_loss}) confirms that this scaled budget provides high-quality targets, allowing the self-improvement loop to bootstrap successfully even on larger grids. The fact that the agent reliably reaches a $100\%$ win rate (Fig.~\ref{fig:winrate_vs_random}) proves that the noise ceiling is successfully pierced by scaling the budget correctly.

\subsection{Model-Free vs. Search-Based Agents}
The tournament results (Fig.~\ref{fig:tournament}) provide a clear verdict on credit assignment in connection games. Hex features incredibly sparse rewards and requires long-horizon planning. Standard model-free agents like DQN struggle to propagate the final win/loss signal back through dozens of moves. Introducing a Minimax search on top of DQN improves performance ($72.0\%$ mean win rate), but it remains distinctly inferior to AlphaZero. AlphaZero's superiority stems from its dual-headed architecture acting as a guided heuristic for MCTS, filtering out catastrophic blunders at decision time---a rigid requirement for a game as tactically unforgiving as Hex.

\subsection{Limitations and Practical Implications}
While the agent dominates standard RL baselines, the evaluation is currently limited to these internal benchmarks. The agent operates as a \emph{tabula rasa}---using no Hex-specific domain knowledge such as virtual connections or dead-cell pruning. Therefore, while it easily defeats model-free algorithms, it would likely trail highly specialized, knowledge-rich solver engines like MoHex. However, from a system design perspective, this framework achieves its primary goal: a single codebase and a single structural parameter that reliably self-configures the search and learning loops for any board size between $5\times5$ and $17\times17$ without manual retuning.

\section{Conclusion}
\label{sec:conclusion}

We presented a fully self-contained, knowledge-free AlphaZero framework for Hex, built around a single governing principle: every critical search and learning parameter must scale automatically with the board dimension. By deriving the MCTS simulation budget, network capacity, and self-play volume directly from the board size, we eliminated the silent failure modes that typically plague self-play on expanding action spaces. 

Supported by a compiled C++ search kernel and hardware-optimized batching, the agent efficiently scales up to larger boards on a single GPU. Our evaluation proves that this dynamic scaling is not merely an optimization, but a strict requirement for stable learning. Furthermore, tournament evaluations demonstrate that this search-based approach decisively outclasses traditional model-free reinforcement learning baselines, achieving a $97.0\%$ tournament win rate.

Future work should focus on closing the absolute strength gap to specialist engines. This can be achieved by integrating modern search enhancements such as Gumbel-AlphaZero action selection and playout caps, as well as benchmarking head-to-head Elo ratings against established, knowledge-rich Hex solvers. Ultimately, the size-scaling principle utilized here offers a robust template for developing RL agents that autonomously adapt to environments with variable complexities.

\section{Implementation notes}
\label{app:impl}
The agent interface is a single callable \texttt{agent(board, action\_set)} returning a
$(\text{row},\text{col})$ move. On the first move of an empty board the agent plays a random
interior cell, avoiding the outer ring whose corner/edge openings are weak in Hex; all
subsequent moves use the full inference-time MCTS budget $2N(n)$. Models are persisted per
board size with their channel and block counts so they reload with the correct architecture.
Self-play workers run network inference on CPU to avoid contending with the GPU training
process, communicating weights and samples through multiprocessing queues with the
file-system tensor-sharing strategy to avoid file-descriptor exhaustion. The repository also
contains model-free baselines---tabular Q-learning, DQN (with an optional minimax variant),
REINFORCE, and PPO---behind the same interface for comparison.

\section{Reproducibility}
\label{app:repro}
The complete source code is publicly available at
\url{https://github.com/philippchn/RL-Hexgame}.
Training is launched via \texttt{python train.py --method alphazero --size N
--iterations K}, with all size-dependent quantities defaulting to the rules of
Section~\ref{sec:scaling} unless overridden. A guard warns when the requested parallel
leaf count exceeds $N/7$, which would make the search too shallow to learn. Checkpoints are
written every $20$ iterations, enabling resumption at a higher simulation budget as used for
the extended $13\times13$ run.
